Änderungen in NI-DAQ 6.9.3 zu NI-DAQ 6.9.2

Der Zweck dieses NI-DAQ-Updates ist zum einen die Unterstützung für neue Geräte hinzuzufügen und zum anderen die Beseitigung von Fehlern der älteren Version.

Unterstützung neuer Geräte

  • NI 6013 für PCI

  • NI 6014 für PCI

Es wurde ein Fehler in DIO Clear (DIG_Block_Clear) für das NI 6534 beseitigt

Wenn Sie das VI DIO_zurücksetzen (oder die Funktion DIG_Block_Clear) aufrufen, schalten die Ports des NI 6534 nicht mehr intermittierend auf andere Zustände um. Der Status verbleibt unverändert auf dem zuletzt ein- oder ausgegebenen Wert, bis das Gerät zurückgesetzt wird.

Fehler in Zusammenhang mit dem NI 6120 behoben

  • Das NI 6120 kann nun mit der maximalen Rate abgetastet werden.

  • Verschiedene Kalibrierungs-Probleme wurden beseitigt.

  • Es wurde für das NI 6120 die Unterstützung für den 12-Bit-Analogtrigger-DAC hinzugefügt.

Unterstützung des Onboard-Erfassungsspeichers für NI 6115 und NI 6120

NI-DAQ 6.9.3 unterstützt mit LabVIEW einfach gepufferte Erfassungen mit 16 oder 32 MS bei Verwendung des Onboard-Speichers. Weitere Informationen finden Sie in der Knowledge Base 2MBBRQB5, “Using NI 6115 and NI 6120 Onboard Memory to Store Acquisition Data for Later Retrieval,”. Dieser Modus unterstützt keine Vortrigger-Erfassungen.

Fehler in Zusammenhang mit dem NI 6052E behoben

Mit NI-DAQ 6.9.3 wird ein in NI-DAQ 6.9.2 existierendes Problem behoben, welches die Erzeugung eines Fehlers verursacht, wenn das NI 6052E bei seiner maximalen Abtastrate betrieben wird.

LabWindows/CVI unterstützt nun längere Namen bei virtuellen Kanälen

NI-DAQ 6.9.3 erlaubt jetzt die Verwendung von mehr als 100 Zeichen langen virtuellen Kanalnamen bei den Easy-I/O-Funktionen.

Problem bei virtuellem Dehnungs-Messkanal des SCXI-1520 behoben

NI-DAQ 6.9.3 behebt das Problem des SCXI-1520 bei der Verwendung des Dehnungs-Messkanals in NI-DAQ 6.9.2. Es kann jetzt ein Nullabgleich des Offsets am Dehnungs-Messkanal des SCXI-1520 vorgenommen werden, ohne das ein Rest-Offset verbleibt.

Problem bei der Erkennung des NI 6025E für PXI in Zusammenhang mit PXI-1010 und SCXI-1125 wurde behoben

Das Problem, daß das SCXI-1125 nicht konfiguriert werden konnte, wenn es zusammen mit gewissen Versionen des NI PXI-6025E in einem PXI-1010-Chassis verwendet wurde, wurde behoben. Sie können nun das SCXI-1125 für den Betrieb mit allen Revisionen des NI PXI-6025E konfigurieren.

Problem des NI 4472 bei kontinuierlicher Datenerfassung behoben

NI-DAQ 6.9.3 reagiert nun auch auf spontane Überlauffehler des Puffers, die unter LabVIEW RT und (selten) unter Windows-Betriebssystemen auftreten können. Dieses Problem betrifft kontinuierliche (doppelt gepufferte) Erfassungen am NI 4472. Wenn das Problem auftritt, springt der Scan-Backlog plötzlich auf einen sehr hohen Wert, und die Erfassung wird mit dem Fehler -10846 (Puffer-Überlauf) angehalten.

(Diese DLL liegt in dem gleichen Verzeichnis, wie die LLBs und ersetzt die Code-Interface-Knoten, die zuvor in jedem VI eingebettet waren.)

    •  _daq301.llb

    • _nidaq32.llb

    • dqchnutl.llb

    • misc.llb

    Diese LLBs und die DLL sind mit LabVIEW für Windows 5.x, LabVIEW RT Version 5.1.2 oder höher, BridgeVIEW 2.x und der LabVIEW Student Edition Version 5.x für Windows kompatibel. Diese LLBs sind abwärtskompatibel, das heißt, sie können mit NI-DAQ für Windows Version 5.0 oder später ausgeführt werden. Wenn ein bestimmter Vorgang oder eine bestimmte Funktion nicht unterstützt wird, geben einige VIs beim Ausführen den DAQ-Fehlercode –10242 (functionNotFoundError) aus.

    Ein Vorteil der Änderung der VIs zum Aufrufen von DLL-Funktionen anstatt CINs ist, dass die VI-Ladezeit erheblich verringert wird. Unsere Bewertungstests haben bei Ausführung auf einem Pentium III 500 MHz Dell Precision 610 unter Windows NT 4.0 SP4 eine durchschnittliche Reduzierung der Ladezeit von mindestens 75 Prozent ergeben, nachdem die NIDAQ32.DLL beim Laden des ersten DAQ-VIs in LabVIEW in den Speicher geladen wurde.

    • <LabVIEW>\vi.lib\flatten.lsb (jetzt in LVDAQ.DLL enthalten)

    • <LabVIEW>\daqdrv (dieselben Funktionen sind jetzt in LVDAQ.DLL enthalten)

    Hinweis: Wenn Sie den LabVIEW Application Builder zum Erstellen einer Programmdatei verwenden, die DAQ Library-VIs aufruft, müssen Sie unter Umständen manuell einen Verweis auf die LVDAQ.DLL in das Build-Script einfügen.

    Das Kompilieren und Ausführen von NI-DAQ-Beispielen in Microsoft Visual C++

    Zum Kompilieren und Ausführen der Beispiele führen Sie folgende Schritte aus:

    1. Öffnen Sie unter Datei»Neu ein neues Projekt. Es erscheint die Registerkarte Projekte (Standardeinstellung).
    2. Wählen Sie Win32-Konsolenanwendung als Projekttyp und geben Sie den gewünschten Projektnamen ein.
    3. Wählen Sie Ein leeres Projekt. Nachdem Sie ein das neue Projekt angelegt haben, sehen Sie im linken Fenster das Datei-Menü und rechts das Datei-Fenster. Klicken Sie auf eine beliebige Datei Ihres Projekts, um diese im rechten Fenster anzeigen zu lassen. Zur besseren Orientierung können Sie von der Ansicht Klassen auf die Ansicht Dateien wechseln.
    4. Wählen Sie Projekt»Dem Projekt hinzufügen»Dateien.
    5. Fügen Sie Ihrem Projekt aus dem Verzeichnis NI-DAQ\LIB die Dateien nidaq32.lib und nidex32.lib hinzu. Außerdem sind die Dateien nidaqex.h und nidaq.h aus dem Verzeichnis NI-DAQ\Include dem Projekt hinzuzufügen. Diese Bibliothek- und Header-Dateien integrieren den Sourcecode für fast alle Beispiele zur Datenerfassung.
    6. Wählen Sie aus dem Menü Extras»Optionen die Registerkarte Verzeichnisse.

    a.       Setzen Sie die Option auf Include-Dateien. Sie müssen auf das gleiche Verzeichnis verweisen wie in NI-DAQ\Include.

    b.      Wählen Sie die Option Bibliothekdateien und verweisen Sie auf den Pfad NI-DAQ\LIB.

    1. Wenn Sie Anwendungen mit Easy-I/O Funktionen benutzen, müssen Sie zusätzliche Dateien einbeziehen, da Easy-I/O Funktionen einige Funktionen aus den CVI Run-Time Bibliotheken benötigen. Es handelt sich dabei um die Bibliotheken easyio.lib, cvisupp.lib, cvirt.lib und cviwmain.lib, die sich im Verzeichnis CVI\EXTLIB befinden.

    a.    Wählen Sie aus dem Menü Extras»Optionen die Registerkarte Verzeichnisse.

    b.   Setzen Sie die Option auf Include-Dateien. Verweisen Sie auf das Verzeichnis CVI\INCLUDE.

    c.    Wählen Sie die Option Bibliothekdateien und verweisen Sie auf den Pfad CVI\EXTLIB.

    1. Sie können dem gewünschten Projekt bei Bedarf die Sourcecode- und Header-Dateien hinzufügen.

    Sie können jetzt aus dem Projekt eine Programmdatei erstellen.

    Hinweise zu DAQPad-6020E

    Extern zeitgesteuertes A/D: Der DAQPad-6020E unterstützt bei geringen Geschwindigkeiten die externe Zeitsteuerung des Abtast- und Scanintervalls (bis zu ca. 90 Hz). Bei schnellerer externer Abtastung führen Sie DAQ-Geräteinformation einstellen (oder einen ähnlichen Vorgang) mit folgenden Einstellungen aus: Informationstyp = Interrupt-Mode und Informationseinstellung = Nach jedem halben FIFO Interrupt generieren.

    Analoges Software-Triggering: Das DAQPad-6020E unterstützt analoge Software-Triggerung nur bei Erfassungen mit geringer Geschwindigkeit (bis zu etwa. 75 Abtastungen/s). Diese Beschränkung gilt nicht für andere Trigger-Arten. Weitere Informationen zu analogem Triggern finden Sie unter dem Thema Config_DAQ_Event_Message in der NI-DAQ Function Reference Help oder im VI AI Trigger Config der LabVIEW DAQ-Hilfe.

    Fortlaufende AI-Übertragungen mit geringer Geschwindigkeit: Bei dem DAQPad-6020E gibt es beim Starten einer fortlaufenden AI-Übertragung mit geringer Geschwindigkeit (von 80 bis zu etwa. 400 Abtastungen/Sek.) eine erhebliche Verzögerung, bevor Daten im Erfassungspuffer eintreffen. Der Grund dafür ist, dass, auch wenn A/D-Wandlungen generiert werden, das Gerät erst dann mit der Übertragung von Daten an den PC beginnt, wenn sein AI-FIFO mindestens halb voll ist.

    Zeitgesteuertes, nicht gepuffertes AI: Auf dem DAQPad-6020E sind zeitgesteuerte, nicht gepufferte AI-Vorgänge, wie im LabVIEW-Beispiel Cont Acq&Chart (hw timed) VI dargestellt, auf etwa. 50 Hz beschränkt. Bei größeren Übertragungsraten reagiert LabVIEW möglicherweise nicht mehr, da NI-DAQ versucht, den AI-FIFO des Gerätes durch Lesen eines Scans pro Zeit zu entleeren.

    Informationen zu NI 6534

    Unterstützte neue Funktionen

    Verbesserte gepufferte Ausgabevorgänge: Die 6534-Geräte verfügen über zwei integrierte Speichermodule, eines für jede Gruppe. Standardmäßig lädt NI DAQ vor dem Starten der Ausgabe die SCARABs bereits entweder mit dem gesamten Datenpuffer oder dem gesamten Speichermodul, je nachdem was kleiner ist. Auf diese Weise werden die Beschränkungen der Busbandbreite aufgehoben. Sie können den Standard-Ladevorgang aber auch übergehen:

    • Führen Sie in LabVIEW das VI DIO-Parameter mit den Einstellungen Parametername = SCARABs Preload Enable und Eingangs-Wert = Aus aus.

    • Wenn Sie in C programmieren, rufen Sie Set_DAQ_Device_Info (NI-DAQ API) mit infoType = ND_FIFO_TRANSFER_COUNT und infoValue = ND_NONE auf.

    Fortlaufende Hochgeschwindigkeitsausgabe mit sich wiederholenden Daten: Die 6534-Geräte unterstützen den Schleifenmodus des Ausgabespeichers, bei dem die Ausgabedaten nur einmal in den SCARAB geladen werden und der gesamte Puffer ständig wiederholt wird, vorausgesetzt, die Puffergröße ist kleiner oder gleich der integrierten Speichergröße:

    • Führen Sie dazu in LabVIEW das VI DIO-Parameter mit der Einstellung Parametername = Pattern Generation Loop Enable aus und führen Sie dann die normale DIO-Konfiguration für fortlaufende Ausgabevorgänge mit aktiviertem Modus für kontinuierliche Erzeugung durch.

    • Wenn Sie in C programmieren, rufen Sie Set_DAQ_Device_Info mit infoType = ND_PATTERN_GENERATION_LOOP_ENABLE und infoValue = ND_ON auf.

    Größe des integrierten Speichers: Die 6534-Geräte bieten Speichermodule in verschiedenen Größen. So können Sie die Speicherkapazität Ihres 6534 Gerätes abfragen:

    • Führen Sie in LabVIEW das VI Geräteinformation lesen [Get DAQ Device Information] mit Informationstyp = SCARAB A Memory Size (für Gruppe 1) oder Scarab B Memory Size (für Gruppe 2) aus.

    • Wenn Sie eine andere Entwicklungsumgebung verwenden, rufen Sie Get_DAQ_Device_Info (in NI-DAQ API) mit infoType = ND_SCARAB_A_MEMORY_SIZE (für Gruppe 1) oder ND_SCARAB_B_MEMORY_SIZE (für Gruppe 2) auf.

    Die Speichergröße wird in Byte ausgegeben.

    20 MHz Muster I/O: Sie können die Taktfrequenz auf 20 MHz einstellen, indem Sie das VI Konfiguration Digital-Takt [Digital Clock Config.vi]oder DIO konfigurieren [DIO Config.vi] ausführen. Wenn Sie in C programmieren, rufen Sie DIG_Block_PG_Config mit Zeitbasis = -3 und Intervall = 1 auf.

    Interrupt-Leistung und -Probleme zwischen PCI-Geräten und Windows 98

    Auf Systemen, die alle nachfolgenden Bedingungen erfüllen, erfolgt die Interrupt-Verarbeitung möglicherweise langsamer:

    • Der Computer verfügt über USB-Anschlüsse.

    • Der Computer läuft unter Windows 98.

    • Sie verwenden ein PCI-Datenerfassungsgerät.

    • Die Datenerfassungsanwendung führt eine Interrupt-intensive Operation aus, beispielsweise Handshaking oder Nachrichtenübermittlung.

    • Der Interrupt Level (IRQL) des PCI-Geräts stimmt mit dem IRQL des USB-Host-Controllers überein (siehe nachfolgenden Abschnitt Ermitteln des IRQL Ihres PCI DAQ-Geräts und USB-/IEEE 1394-Host-Controllers).

    Extreme Langsamkeit, Timeouts oder Blockierungen können auf Systemen auftreten, auf die exakt alle folgenden Bedingungen zutreffen:

    • Der Computer verfügt über einen IEEE 1394-Adapteranschluss.

    • Der Computer läuft unter Windows 98.

    • Sie verwenden ein PCI-Datenerfassungsgerät.

    • Die Datenerfassungsanwendung führt eine Interrupt-intensive Operation aus, beispielsweise Analoges SW-Triggering, asynchrone Occurences oder andere Nachrichtenübermittlungsbeispiele.

    • Der Interrupt-Level (IRQL) des PCI-Geräts stimmt mit dem IRQL des IEEE 1394-Host-Controllers überein (siehe nachfolgenden Abschnitt Ermitteln des IRQL Ihres PCI DAQ-Geräts und USB-/IEEE 1394-Host-Controllers).

    Ermitteln des IRQL Ihres PCI DAQ-Geräts und USB-/IEEE 1394-Host-Controllers

    Gehen Sie folgendermaßen vor, um zu ermitteln, ob der IRQL Ihres PCI DAQ-Geräts und USB-/IEEE 1394-Host-Controllers übereinstimmen:

    1. Klicken Sie mit der rechten Maustaste auf das Symbol Arbeitsplatz. Es befindet sich meist oben links auf dem Bildschirm.
    2. Wählen Sie im Popup-Menü die Option Eigenschaften, um das Fenster für die System-Eigenschaften zu öffnen.
    3. Klicken Sie auf das Register Geräte-Manager.
    4. Doppelklicken Sie auf Computer, um das Fenster Eigenschaften von Computer zu öffnen.
    5. Vergewissern Sie sich, dass die Option Interrupt (IRQ) aktiviert ist.
    6. Blättern Sie durch das Auswahlfeld bis zum PCI DAQ-Gerät und notieren Sie die Nummer in der Spalte Einstellung (den IRQ für das Gerät).
    7. Blättern Sie durch das Auswahlfeld bis zum USB-/IEEE 1394-Host-Controller und notieren Sie den IRQ in der Spalte Einstellung.
    8. Vergleichen Sie die Nummer, die Sie für Ihr PCI DAQ-Gerät notiert haben, mit der Nummer, die Sie für den USB-/IEEE 1394-Controller notiert haben.

    Wenn die Einstellungsnummer des PCI-Datenerfassungsgerät mit dem USB-/IEEE 1394-Host-Controller übereinstimmt, gehen Sie entsprechend den nachfolgenden Anweisungen vor, um die Interrupt-Verarbeitungsleistung zu verbessern.

    Hinweis: Die gemeinsame Nutzung eines IRQL zwischen dem USB-/IEEE 1394-Host-Controller und dem PCI-Datenerfassungsgerät betrifft nur die Geschwindigkeit der Verarbeitung von Interrupt-Transaktionen. DMA oder andere I/O-Vorgänge sind nicht betroffen. Ebenso sollte die gemeinsame Nutzung eines IRQL zwischen einer beliebigen Anzahl von PCI-Datenerfassungsgeräten oder anderen PCI-Geräten, beispielsweise einem Video- oder Netzwerkadapter, die Geschwindigkeit der Interrupt-Verarbeitung nicht beeinträchtigen.

    USB- und IEEE 1394-Unterstützungsanforderungen und Beschränkungen

    Betriebssystemunterstützung für USB und IEEE 1394

    Seit NI-DAQ Version 6.7 wird die Software-Unterstützung für USB und IEEE 1394 nur noch für die Einzelprozessorsysteme Windows 2000, XP, Me und 98 zur Verfügung gestellt. Wenn Sie eine andere Windows-Version verwenden, müssen Sie Ihr Betriebssystem für die Verwendung von USB- oder IEEE 1394-Geräten aktualisieren.

    National Instruments Software-Unterstützung für USB und IEEE 1394

    • Für die Verwendung von USB-DAQ-Geräten ist NI-DAQ-Version 6.5 oder später erforderlich.

    • Für die Verwendung von IEEE 1394 DAQ-Geräten ist NI-DAQ Version 6.6 oder später erforderlich.

    • Für die Verwendung von USB oder IEEE 1394 ist LabVIEW Version 4.0 oder später erforderlich.

    Mögliche Probleme

    Es ist am besten, wenn Sie Ihr USB- oder 1394-DAQPad-Gerät an den PC anschließen und einschalten bevor Sie Ihre DAQ-Applikation oder den MAX starten. Wenn Sie das DAQPad einschalten, nachdem die DAQ-Anwendung geladen wurde, setzen Sie das Gerät in der Software zurück, bevor Sie versuchen, es durch Ausführen des Device Reset.vi (in LabVIEW) oder durch Aufrufen von Init_DA_Brds() im NI-DAQ API zu starten.

    Wenn Sie eine DAQ-Anwendung verwenden, die keine manuellen Rücksetzungen unterstützt, beispielsweise VirtualBench, müssen Sie das USB- oder IEEE 1394-DAQPad anschließen und Einschalten, bevor Sie die Anwendung ausführen.

    In seltenen Fällen werden USB- oder IEEE 1394-Geräte nicht in der Geräte-Baumstruktur des Measurement & Automation Explorer angezeigt, wenn Fenster des Windows Explorer (auch Internet Explorer) geöffnet sind. Wenn dieses Problem auftritt, schließen Sie alle Explorer-Fenster und aktualisieren Sie die Ansicht des Measurement & Automation-Fensters durch Drücken von <F5>. Bleibt das Problem weiter bestehen, so starten Sie das System neu.

    USB- und IEEE 1394-DAQPads unterstützen den Windows 2000/XP/Me/98-Unterdrückungsmodus nicht.

    Konfiguration und Energieverwaltung

    USB- und IEEE 1394-Geräte unterstützen Plug and Play-Konfigurationen und den Geräteaustausch bei laufendem Computer und Windows Me/98 bietet eine Betriebssystemunterstützung für die Energieverwaltung. Wenn Ihre DAQ-Anwendung aktiv sein muss, während der Computer unbeaufsichtigt ist, deaktivieren Sie die Energieverwaltungsfunktionen im Betriebssystem und im BIOS und den Bildschirmschoner.

    Leistung

    Die Leistung von USB/IEEE 1394 hängt stark von systemspezifischen Faktoren wie der CPU-Geschwindigkeit, der Speicherarchitektur und den System-Chipsatzeigenschaften ab. USB- und IEEE 1394-Vorgänge sind Software-intensiv. Aus diesem Grund kann das Ausführen anderer Anwendungen während der Datenerfassung die Leistung beeinträchtigen. Hochgeschwindigkeits-USB- und IEEE 1394-Vorgänge können auch die Leistung anderer Anwendungen beeinträchtigen.

    Alle an Ihren PC angeschlossenen USB- und IEEE 1394-Geräte nutzen eine gemeinsame Bandbreite. Die gleichzeitige Verwendung mehrerer USB- und IEEE 1394-Geräte kann die Leistung von DAQ-Anwendungen beeinträchtigen. Seitdem sich die USB- und IEEE 1394-Host-Controller des PCs auf dem PCI-Bus befinden, kann auch die Verwendung anderer, bandbreitenintensiver PCI-Geräte die Leistung beeinträchtigen.

    Da Daten paketweise über die USB- und IEEE 1394-Busse übertragen werden, ist die Zeitplanung bei Übertragungen kleiner oder präziser Datenmengen weniger bestimmt als bei Plug-In-DAQ-Geräten. So ist die Abweichung im Timing möglicherweise bei Nachrichtenübermittlungen und Interrupt-intensiven Vorgängen für USB- und IEEE 1394-Geräte größer.

Nachdem Sie dieses Passwort festgelegt haben, müssen Clients, die eine Remote-Konfiguration des Servers über den MAX durchführen möchten (durch Klicken auf die Optionen Werkzeuge»NI-DAQ-Konfiguration»RDA-DAQ-Konfiguration), zunächst das richtige Passwort eingeben. Clients, denen das richtige Root-Passwort bekannt ist, haben die Möglichkeit, über die Remote DAQ-Konfiguration sowohl das Root-Passwort des Servers, als auch alle einzelnen Geräte-Passwörter zu ändern.

    Bei NI-DAQ 6.7 und höheren Versionen kann RDA jetzt auch eine Zeitüberschreitung erfassen. Dies ist dann nützlich, wenn die Netzwerkverbindung verloren geht, während der RDA-Client auf eine Antwort vom RDA-Server wartet. Ohne ein Timeout wartet der RDA-Client ohne Zeitbeschränkung, so dass das LabVIEW-Programm scheinbar nicht mehr reagiert. Mit der Timeout-Funktion gibt das LabVIEW-Programm stattdessen den Fehler -10340 noConnectError aus.

    Der Standardwert für dieses Timeout ist 240.000 Millisekunden (4 Minuten). Wenn Sie diesen Wert ändern möchten, müssen Sie einen Wert zum vorhandenen Registrierungsschlüssel hinzufügen. Gehen Sie dazu folgendermaßen vor:

    1. Führen Sie das Programm regedit.exe oder regedt32.exe aus.
    2. Öffnen Sie HKEY_LOCAL_MACHINE»SOFTWARE»National Instruments»NI-DAQ»CurrentVersion»Konfigurationsmanager.
    3. Wählen Sie im Bearbeitungsmenü die Optionen New»DWORD Value.
    4. Geben Sie dem neuen Wert den Namen ReceiveTimeoutValue.
    5. Doppelklicken Sie auf ReceiveTimeoutValue, um ein Dialogfeld zu öffnen, in das Sie die Daten des Werts eingeben . Geben Sie das gewünschte Timeout in Millisekunden ein. Stellen Sie sicher, dass die Basis auf Dezimal eingestellt ist. Die Angabe eines Timeout-Werts von 0 bedeutet ein unendliches Timeout, was zu demselben Verhalten wie bei den NI-DAQ-Versionen vor Version 6.7 führt.
    6. Schließen Sie das Programm regedit.exe oder regedt32.exe.

    Beachten Sie, dass NI-DAQ-Aufrufe, die Daten lesen oder schreiben, je nach den Übertragungsraten und der Datenmenge lange Zeit in Anspruch nehmen können. Der Aufruf des VI AI - Puffer lesen und das Lesen von 100 Scans bei einer Frequenz von 10 Scans/s dauert 10 s. Wenn der ReceiveTimeoutValue kleiner als 10.000 ms ist, erhalten Sie den Fehler –10340, auch wenn die Netzwerkverbindung in Ordnung ist.

    Das Weitergeben von Unsigned 16- und 32-Bit Integer-Parametern in Visual Basic

    Vorzeichenlose 16-Bit-Parameter: Visual Basic INTEGERS sind vorzeichenbehaftete Werte. Alle Werte über 32767 werden als negative Werte interpretiert. Dieses Problem lässt sich wie folgt umgehen:

    1. Wenn die Funktion einen vorzeichenlosen 16-Bit-Integer ausgeben soll, Sie aber einen negativen Wert zurückerhalten, addieren Sie 65536 zu diesem Wert.

    Beispiel:

    Dim myUnsignedCnt As Long

    Dim retCnt As Integer                             

    ‘(retCnt ist der Wert, der von der aufgerufenen Funktion ausgegeben wird)

    ‘Sie können eine DAQ-Funktion aufrufen, die einen vorzeichenlosen 16-Bit Wert zurückgibt und diesen in retCnt speichert.

    if retCnt < 0 then

    myUnsignedCnt = CInt(retCnt) + 65536

    else

    myUnsignedCnt = retCnt

    end if

    1. Um einen vorzeichenlosen 16-Bit-Wert an eine Funktion mit einem vorzeichenlosen 16-Bit Typparameter weiterzugeben, müssen Sie den Wert zunächst berechnen.

    Beispiel:

    Dim myUnsignedCnt As Long

    Dim ToPassVal As Integer

    ‘Sie berechnen den weiterzugebenden Wert und speichern ihn in myUnsignedCnt.

    if myUnsignedCnt > 32767 then

    ToPassVal = CInt(myUnsignedCnt - 65536)

    else

    ToPassVal = CInt(myUnsignedCnt)

    end if

    Vorzeichenlose 32-Bit-Parameter: Visual Basic LONG Integertypen sind vorzeichenbehaftete 32-Bit-Typen. Alle Werte über 2.147.483.647 werden als negative Werte interpretiert. Dieses Problem lässt sich wie folgt umgehen:

    1. Wenn die Funktion einen vorzeichenlosen 32-Bit Integer ausgeben soll, Sie aber einen negativen Wert zurückerhalten, addieren Sie 4.294.967.296 zu diesem Wert.

    Beispiel:

    Dim myUnsignedCnt As Double

    Dim retCnt As Long       

    ‘(retCnt ist der Wert, der von der aufgerufenen Funktion ausgegeben wird)

    ‘Sie können eine DAQ-Funktion aufrufen, die einen vorzeichenlosen 16-Bit Wert zurückgibt und diesen in retCnt speichert.

    if retCnt < 0 then

    myUnsignedCnt = CInt(retCnt) +4294967296

    else

    myUnsignedCnt = retCnt

    end if

    2.      Um einen vorzeichenlosen 32-Bit-Wert an eine Funktion mit einem vorzeichenlosen 32-Bit Typparameter weiterzugeben, müssen Sie den Wert zunächst berechnen.

    Beispiel:

    Dim myUnsignedCnt As Double

    Dim ToPassVal As Long

    ‘Sie berechnen den weiterzugebenden Wert und speichern ihn in myUnsignedCnt.

    if myUnsignedCnt > 2147483647 then

    ToPassVal = CLng(myUnsignedCnt - 4294967296)

    else

    ToPassVal = CLng(myUnsignedCnt)

    end if
