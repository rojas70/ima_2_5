Modifications apportées entre NI-DAQ 6.9.2 et NI-DAQ 6.9.3

Le but de cette version est d'ajouter du support pour de nouveaux périphériques et de remédier à certains problèmes rencontrés sous la précédente version de NI-DAQ.

Nouveaux périphériques supportés

  • NI 6013 pour PCI

  • NI 6014 pour PCI

Problème résolu pour DIO Ré-initialiser (DIG_Block_Clear) avec NI 6534

Lorsque vous appelez le VI DIO Ré-initialiser (DIO Clear.VI) (ou la fonction DIG_Block_Clear), les ports sur le périphérique NI 6534 ne basculeront plus sur un autre état de façon intermittente. L'état reste inchangé depuis la dernière valeur ayant été transmise ou retournée jusqu'à ce que le périphérique soit réinitialisé.

Problèmes résolus avec le périphérique NI 6120

  • Les périphériques NI 6120 peuvent maintenant être échantillonnés à leur vitesse maximale.

  • Plusieurs problèmes d'étalonnage ont été résolus.

  • Le support pour le C N/A à déclenchement analogique 12 bits a été ajouté pour le NI 6120.

Support pour mémoire d'acquisition embarquée sur les périphériques NI 6115 et NI 6120

NI-DAQ 6.9.3 supporte avec LabVIEW les acquisitions à simple buffer de 16 ou 32 mégaéchantillons vers la mémoire embarquée. Veuillez consulter l'entrée 2MBBRQB5 de la Knowledge Base (base de connaissances), “Using NI 6115 and NI 6120 Onboard Memory to Store Acquisition Data for Later Retrieval,” pour obtenir plus d'informations Ce mode ne supporte pas les acquisitions à pré-déclenchement.

Résolution du problème rencontré avec le périphérique NI 6052E

Par rapport à NI-DAQ 6.9.2, NI-DAQ 6.9.3 résout le problème qui générait une erreur lors de l'utilisation du périphérique NI 6052E à sa vitesse d'échantillonnage maximale.

Noms de voies virtuelles plus longs supportés avec LabWindows/CVI

NI-DAQ 6.9.3 autorise l'utilisation de noms de voies virtuelles possédant plus de 100 caractères avec les fonctions d'E/S simples de LabWindows/CVI. 

Résolution du problème lié aux voies virtuelles pour mesures de contrainte rencontré avec le SCXI-1520

NI-DAQ 6.9.3 résout le problème lié aux voies virtuelles pour mesures de contrainte rencontré avec NI-DAQ 6.9.2.

Résolution du problème de reconnaissance rencontré avec le périphérique NI 6025E pour PXI, le PXI-1010 et le SCXI-1125

Le problème du SCXI-1125 qui ne peut pas être configuré lorsqu'il est utilisé avec certaines versions du périphérique NI PXI-6025E dans un châssis PXI-1010 a été résolu. Vous pouvez maintenant configurer le SCXI-1125 pour un fonctionnement avec toutes les révisions du périphérique NI PXI-6025E.

Résolution du problème rencontré lors d'acquisitions continues avec le périphérique NI 4472

NI-DAQ 6.9.3 traite une erreur de débordement de buffer qui peut se produire avec LabVIEW RT et (rarement) avec les systèmes d'exploitation Windows. Le problème affecte des acquisitions continues (à buffer circulaire) sur le périphérique NI 4472 pour PCI et PXI. Lorsque ce problème se manifeste, le nombre de balayages restants monte brusquement et l'acquisition s'interrompt avec une erreur -10846 de débordement de buffer.

    Changements affectant les bibliothèques DAQ de LabVIEW pour Windows pour NI-DAQ 6.7 et les versions ultérieures

    Depuis NI-DAQ 6.7, les bibliothèques d'acquisition de données de LabVIEW pour Windows, mentionnées ci-dessous et installées dans le répertoire <LabVIEW> appellent désormais les fonctions d'une DLL (Dynamic Link Library) appelée LVDAQ.DLL. Cette DLL se trouve dans le même répertoire que les LLB et remplace les CIN (Code Interface Nodes) qui étaient jusqu'alors incorporés dans chaque VI.

    •  _daq301.llb

    • _nidaq32.llb

    • dqchnutl.llb

    • misc.llb

    • zadvd.llb

    • zadvdctr.llb

    Ces LLB sont compatibles avec LabVIEW 5.x pour Windows , LabVIEW RT version 5.1.2 ou plus récente, BridgeVIEW 2.x, ainsi qu’avec LabVIEW Student Edition version 5.x pour Windows. Ces LLB assurent une compatibilité descendante, c’est-à-dire qu’elles peuvent être chargées sous NI-DAQ pour Windows version 5.0 ou plus récente sans que le bouton Exécution n’apparaisse brisé. Si une opération ou fonction particulière n'est pas supportée, certains VIs renvoient un code derreur DAQ –10242 (functionNotFoundError) quand vous les exécutez.

    L’intérêt de l’appel d’une DDL en lieu et place des CIN est que le temps de chargement de ces VIs est amélioré de manière drastique. Nos évaluations de performance, effectuées sur un Pentium III500 MHz Dell Precision 610 tournant sous Windows NT 4.0 SP4, ont montré une réduction du temps de chargement d’au moins 75% en moyenne, après le chargement initial de la bibliothèque NIDAQ32.DLL lors du premier chargement d’un VI DAQ sous LabVIEW.

    Les fichiers suivants ne sont plus nécessaires à LabVIEW pour Windows lorsque sont utilisées les LLB d'acquisition de données installées par la version 6.7 ou une version ultérieure de NI-DAQ :

    • <LabVIEW>\vi.lib\flatten.lsb (maintenant compris dans LVDAQ.DLL)

    • <LabVIEW>\daqdrv (même fonctionnalité maintenant comprise dans LVDAQ.DLL)

    Remarque :   Si vous utilisez l'Application Builder pour créer un exécutable qui appelle des VIs des bibliothèques DAQ, vous aurez peut-être à ajouter manuellement une référence à LVDAQ.DLL dans votre script de construction.

    Compilation et exécution d'exemples NI-DAQ sous Microsoft Visual C++

    Pour compiler et exécuter les exemples, veuillez suivre les étapes suivantes :

    1. Ouvrez un nouveau projet via le menu File»New. L’onglet Projects devrait apparaître par défaut.
    2. Sélectionnez Win32 Console Application comme type de projet et donnez un nom à votre projet.
    3. Sélectionnez Empty Application. Lorsque vous créez un nouveau projet, votre structure de répertoire apparaît sur la gauche et les fichiers sur la droite. Lorsque vous double-cliquez sur n’importe quel fichier du projet, le fichier s’affiche sur la partie droite. Pour faciliter votre navigation, basculez de Class View en File View.
    4. Lancez Project»Add to Project»File.
    5. Sélectionnez le répertoire NI-DAQ\LIB et ajoutez les fichiers nidaq32.lib et nidex32.lib. Insérez également les fichiers nidaqex.h et nidaq.h à partir du répertoire NI-DAQ\Include. L’ajout de ces deux bibliothèques et fichiers header intègre le code source de la plupart des exemples d’applications d’acquisition de données.
    6. Allez dans Tools»Options et sélectionnez l’onglet Directories.

    a.       Activez l’option Include Files. Assurez-vous que vous pointez vers le même répertoire dans NI-DAQ\Include.

    b.      A partir du même onglet, activez l’option Libraries et définissez un chemin pour NI-DAQ\LIB.

    1. Si l’ application utilise des fonctions de type Easy I/O, vous devez inclure des fichiers supplémentaires car ces fonctions utilisent des fonctions des bibliothèques Run-Time de CVI. Les bibliothèques supplémentaires sont easyio.lib, cvisupp.lib, cvirt.lib et cviwmain.lib et sont situées dans le répertoire CVI\EXTLIB.

    a.    Allez dans Tools»Options et sélectionnez l’onglet Directories.

    b.    Activez l’option Include Files. Pointez vers le répertoire CVI\INCLUDE.

    c.    A partir du même onglet, activez l’option Libraries et définissez un chemin pour CVI\EXTLIB.

    1. Ajoutez le code source et le header si nécessaire dans le projet que vous utilisez.

    Vous devriez pouvoir créer un exécutable à partir du projet.

    Remarques sur le DAQPad-6020E

    Conversion A/N cadencée par une source externe : Par défaut et pour des faibles vitesse (<90Hz), les horloges d’échantillonnage et de balayage du DAQPad-6020E sont externes. Pour des vitesses supérieures, exécuter le VI Définir les infos d’un périphérique DAQ (ou un équivalent) comme suit : type d'information = mode de génération d'interruption et paramètre d'information = générer une interruption toutes les demi-FIFOs.

    Déclenchement analogique logiciel : Cette fonction n'est possible avec le DAQPad-6020E que pour des acquisitions à faible vitesse (jusqu'à environ 75 échantillons/s). Cette restriction ne s'applique pas aux autres types de déclenchement. Pour plus d'informations sur le déclenchement analogique, reportez-vous au sujet Config_DAQ_Event_Message de l'aide en ligne NI-DAQ Function Reference ou à AI Trigger Config.vi (AI-Configurer un déclenchement) dans l’Aide LabVIEW .

    Transferts en continu de données acquises à faibles vitesse : Lorsque le DAQPad-6020E commence à acquérir des données à faible vitesse (entre environ 80 et 400 échantillons/s), un délai notable s'écoule avant que les premières données n'arrivent dans le buffer d'acquisition. La raison en est que, bien que des conversions A/N aient commencé, le périphérique ne transfère pas de données vers l’ordinateur avant que sa FIFO ne soit au moins à moitié pleine.

    AI cadencé non bufférisé : Sur le DAQPad-6020E, les opérations AI cadencées non bufférisées, comme illustrées dans le VI d’exemple de LabVIEW Cont Acq&Chart (hw timed), sont limitées à environ 50 Hz. A des vitesses supérieures, LabVIEW peut se figer lorsque le driver NI-DAQ tente de vider la FIFO d’acquisition analogique en lisant un balayage à la fois sur le périphérique.

    Informations sur la famille NI 6534

    Nouvelles fonctions supportées

    Améliorations des opérations bufferisées : Les périphériques 6534 comprennent deux modules mémoire embarqués sur la carte, un pour chaque groupe. Par défaut et avant de commencer toute opération de sortie, NI-DAQ va précharger les puces SCARABs avec la totalité, soit du buffer de données, soit du module mémoire (c-à-d avec celui de plus petit taille), éliminant ainsi les limitations de la bande passante du bus. Cependant, il vous est possible de contourner le chargement par défaut :

    • Sous LabVIEW, utilisez le VI DIO-Paramètre avec nom de paramètre = Autoriser le préchargement de Scarab et valeur d'entrée= Off.

    • Si vous programmez en C, appelez Set_DAQ_Device_Info avec infoType = ND_FIFO_TRANSFER_COUNT et infoValue = ND_NONE.

    Sortie continue, très haute vitesse, avec données répétitives : Les périphériques 6534 supportent le mode de mémoire en boucle dans lequel les données de sortie sont chargées dans le SCARAB en une seule fois et le contenu du buffer tout entier est généré en boucle, pour autant que la taille du buffer soit inférieure ou égale à celle de la mémoire embarquée sur la carte.

    • Pour cela, exécutez le VI DIO-Paramètre en sélectionnant comme nom de paramètre = Génération en boucle du Pattern autorisée; puis configurez normalement votre E/S numérique pour une opération de sortie continue en autorisant le mode de régénération.

    • Si vous programmez en C, appelez Set_DAQ_Device_Info avec infoType = ND_PATTERN_GENERATION_LOOP_ENABLE et infoValue = ND_ON.

    Taille de la mémoire embarquée sur la carte : Les périphériques 6534 présentent des modules mémoire de différentes tailles. Pour obtenir la taille de la mémoire embarquée sur votre périphérique 6534 :

    • Sous LabVIEW, exécutez le VI Obtenir des infos sur un périphérique DAQ avec type dinformation = Taille mémoire du SCARAB A (pour le groupe 1) ou Taille mémoire du SCARAB B (pour le groupe 2).

    • Si vous utilisez un autre environnement de programmation, appelez Get_DAQ_Device_Info (API NI-DAQ) avec info Type = ND_SCARAB_A_MEMORY_SIZE (pour le groupe 1) ou ND_SCARAB_B_MEMORY_SIZE (pour le groupe 2).

    La taille de la mémoire est donnée en octets.

    E/S de patterns à 20 MHz : Vous pouvez fixer la fréquence d’horloge à 20 MHz en exécutant le VI Configurer une horloge numérique ou le VI DIO-Configurer. Si vous programmez en C, appelez call DIG_Block_PG_Config avec timebase = -3 et interval = 1.

    Problèmes de performance liés aux transferts par interruptions avec les périphériques PCI sous Windows 98

    Le fonctionnement par interruptions peut être lent sur des systèmes remplissant exactement les conditions suivantes :

    • L'ordinateur possède des ports USB.

    • L'ordinateur est équipé de Windows 98.

    • Vous utilisez un périphérique PCI d'acquisition de données.

    • L'application d'acquisition de données exécute de nombreuses opérations par interruptions, telles que handshaking ou messaging

    • Le niveau de requête d’interruption (IRQL) du périphérique PCI correspond à l'IRQL du contrôleur hôte USB (voir ci dessous Détermination de l’IRQL de votre périphérique DAQ PCI et de votre contrôleur hôte USB/1394).

    Une extrême lenteur, des dépassements de limite de temps ou des blocages peuvent se produire sur des systèmes où les conditions suivantes sont toutes réunies :

    • L'ordinateur possède un adaptateur pour port 1394

    • L'ordinateur est équipé de Windows 98.

    • Vous utilisez un périphérique PCI d'acquisition de données.

    • L'application d'acquisition de données exécute de nombreuses opérations par interruptions telles que : déclenchement logiciel analogique, occurrences asynchrones ou autres exemples de messaging

    • Le niveau de requête d’interruption (IRQL) du périphérique PCI correspond à l'IRQL du contrôleur hôte 1394 (voir ci dessous Détermination de l’IRQL de votre périphérique DAQ PCI et de votre contrôleur hôte USB/1394).

    Détermination de l’IRQL de votre périphérique DAQ PCI et de votre contrôleur hôte USB/1394

    Pour déterminer si les IRQL de votre périphérique DAQ PCI et de votre contrôleur hôte USB/1394 sont les mêmes, suivez les instructions suivantes :

    1. Faites un clic droit sur l'icône Poste de travail, généralement située dans le coin supérieur gauche de l'écran.
    2. Sélectionez Propriétes dans le menu pour ouvrir la fenêtre Propriétés système.
    3. Cliquez sur l'onglet Gestionnaire de périphériques (ou onglet Matériel, puis bouton Gestionnaire de périphériques).
    4. Double-cliquez sur Ordinateur pour ouvrir la fenêtre de propriétés de l'ordinateur.
    5. Assurez-vous que l'option Requête d'interruption (IRQ) est sélectionnée.
    6. Faites défiler les paramètres jusqu'à trouver le périphérique PCI (par exemple PCI-6071E) et notez le numéro inscrit dans la colonne Paramètre (l’IRQ pour le périphérique).
    7. Faites défiler les paramètres jusqu’à trouver le contrôleur hôte USB/1394 et notez son IRQ inscrit dans la colonne Paramètre.
    8. Comparez le numéro que vous avez noté pour votre périphérique DAQ PCI avec celui que vous avez noté pour le contrôleur hôte USB/1394

    Si le numéro de paramètre du périphérique PCI d'acquisition de données correspond à celui du contrôleur hôte USB/IEEE 1394, vous pouvez améliorer les performances de votre système lors d’opérations intensives par interruptions en suivant les instructions ci-dessous.

    Remarque :   le partage d'un niveau de requête d'interruption entre le contrôleur hôte USB/IEEE 1394 et le périphérique PCI d'acquisition de données n'affecte que la vitesse des transferts par interruptions. Cela n’affecte pas les transferts par canaux DMA ou autre E/S. D’autre part, le partage d’un IRQL entre plusieurs cartes PCI DAQ ou d’autres périphériques PCI tels que des cartes vidéo ou réseau, ne devrait pas avoir d’effet sur la vitesse des opérations sous interruption.

      1. Double-cliquez sur Contrôleur de bus 1394 (remarque : ce nom peut varier).
      2. Double-cliquez sur le périphérique 1394

    3. Cliquez sur OK pour enregistrer les paramètres et fermez la fenêtre Propriétés système.
    4. Redémarrez votre système

    Conditions requises et limitations pour le support des bus USB et 1394

    Support des bus USB et 1394 par les systèmes d'exploitation

    Depuis NI-DAQ 6.7, le support logiciel des bus USB et IEEE 1394 n'est fourni que pour les systèmes monoprocesseur Windows 2000/XP/Me/98. Si vous travaillez avec une autre version de Windows, vous aurez besoin de mettre à jour votre système d’exploitation pour utiliser les périphériques USB ou 1394.

    Support des bus USB et 1394 par les logiciels National Instruments

    • La version 6.5 ou une version ultérieure de NI-DAQ est requise pour le fonctionnement des périphériques DAQ USB.

    • La version 6.6 ou une version ultérieure de NI-DAQ est requise pour le fonctionnement des périphériques DAQ 1394.

    • La version 4.0 ou une version ultérieure de LabVIEW est requise pour le fonctionnement des périphériques DAQ USB et 1394.

    Problèmes potentiels

    Il est recommandé de connecter votre périphérique DAQPad USB ou 1394 à l'ordinateur et de l'allumer avant de lancer votre application DAQ ou MAX. Si jamais vous connectez ou allumez le périphérique après avoir lancé l'application DAQ, réinitialisez le périphérique par logciel avant de l'utiliser en exécutant le VI Réinitialiser un périphérique dans LabVIEW ou en appelant Init_DA_Brds() dans l'API NI-DAQ.

    Si vous utilisez une application DAQ qui ne supporte pas les réinitialisations manuelles, comme VirtualBench, vous devez connecter et allumer votre DAQPad USB ou 1394 avant d’exécuter votre application

    Lors de rares occasions, les périphériques USB ou 1394 n’apparaîtront pas parmis les périphériques listés dans Measurement & Automation Explorer (MAX) si aucune fenêtre de Windows Explorer (y compris Internet Explorer) n’est ouverte. Si ce problème arrive, fermez toutes vos fenêtres d’Explorer et rafraîchissez la fenêtre Measurement &amp; Automation en appuyant sur la touche F5. Si le problème persiste, redémarrez le système.

    Les DAQPads USB et 1394 ne supportent pas le mode Suspendre de Windows 2000/XP/Me/98.

    Gestion de la configuration et de l'alimentation

    Les périphériques USB et 1394 sont plug-and-play et supportent l’insertion à chaud ; de plus, le système d’exploitation Windows Me/98 fournit des fonctionnalités pour la gestion de l’alimentation. Bien que ces fonctionnalités facilitent généralement la configuration des périphériques, veuillez suivre les conseils donnés dans cette section pour éviter des problèmes de configuration.

    Sous certaines conditions, il se peut que des problèmes surgissent lors d'un débranchement simultané de plusieurs périphériques 1394 au cours d'une acquisition de données. Il est très important de connecter ou déconnecter vos périphériques avant de charger tout logiciel d'acquisition de données.

    Après avoir déconnecté ou éteint un périphérique, il est conseillé d’attendre cinq à dix secondes (ou le temps que le sablier disparaisse) avant de rallumer le périphérique.

    Sur certains ordinateurs, la gestion d'alimentation peut interférer avec les applications DAQ actives qui utilisent des périphériques USB ou 1394. Si vous prévoyez de laisser fonctionner votre application DAQ pendant une longue période, désactivez les fonctionnalités de gestion de l’alimentation dans le système d’exploitation et le BIOS, ainsi que votre écran de veille.

    Pour désactiver la gestion de l'alimentation sous Windows 98, procédez comme suit :

    1. Ouvrez le Panneau de configuration (Cliquez sur Démarrer»Paramètres»Panneau de configuration).
    2. Double-cliquez sur l’icône Gestion de l'Alimentation (options d'alimentation sous Windows XP) puis cliquez sur l’onglet Modes de gestion de l'alimentation.
    3. Dans le menu Modes de gestion de l'alimentation, choisissez Toujours allumé.
    4. Définissez les Paramètres pour le mode de gestion de l'alimentation Toujours allumé comme ci-dessous :
      1. Mise en veille : Jamais (Sous Windows XP, choisissez l’onglet Hibernation et sélectionnez Désactiver l'hibernation.)
      2. Mise en veille du moniteur : Tout choix est acceptable
      3. Arrêt des disques durs : Jamais
    5. Cliquez sur OK pour enregistrer les paramètres et fermer la gestion de l’alimentation.
    6. Fermez le Panneau de configuration

    Pour déterminer si votre système possède une gestion de l’alimentation intégrée au BIOS, ainsi que les instructions pour désactiver les fonctionnalités de gestion de l’alimentation du BIOS, veuillez vous référer à la documentation de votre système.

    Pour désactiver votre écran de veille, procédez comme suit :

    1. Faites un clic droit sur le bureau de Windows et sélectionnez Propriétés dans le menu local pour ouvrir la fenêtre de Propriétés d’affichage.
    2. Cliquez sur l’onglet Ecran de veille.
    3. Dans le menu de l’Ecran de veille, sélectionnez Aucun.
    4. Cliquez sur OK pour enregistrer vos paramètres et sortez de la fenêtre Propriétés d’affichage.

    Performances

    Les performances des bus USB et 1394 dépendent largement de facteurs propres au système tels que la vitesse du processeur, l'architecture de la mémoire et les caractéristiques du chipset du système. Les opérations sur les bus USB et 1394 sont consommatrices de ressources logicielles, et le fait d'exécuter d'autres applications en parallèle peut affecter les performances des opérations d'acquisition de données. Les opérations à haute vitesse sur les bus USB et 1394 peuvent également affecter les performances d'autres applications.

    Tous les périphériques USB et 1394 connectés à votre ordinateur partagent la même bande passante. En conséquence, l'utilisation simultanée de plusieurs périphériques USB et 1394 peut affecter les performances des applications DAQ. En outre, puisque les contrôleurs hôtes USB et 1394 du PC résident sur le bus PCI, l'utilisation d'autres périphériques PCI consommateurs de bande passante peut diminuer les performances.

    Du fait que les données sont transférées par paquets sur les bus USB et 1394, les transferts de petites quantités de données sont moins déterministes qu'avec des périphériques DAQ enfichés dans le PC. Par exemple, les temps d'exécution d'opérations contrôlées par interruptions et messaging sont susceptibles de varier davantage avec des périphériques USB et 1394.

    Accès aux périphériques déportés (RDA)

    Si vous souhaitez utiliser la version 6.9 de NI-DAQ dans une application de type RDA, assurez-vous que cette version de NI-DAQ est installée à la fois sur le client (machine locale) et sur le serveur RDA (machine distante). Cette exigence provient des nouvelles fonctionnalités RDA en matière de sécurité qui ont été ajoutées à cette nouvelle version.

    Les fonctionnalités de sécurité vous permettent de spécifier un mot de passe pour chaque périphérique configuré sur le serveur, ceci dans le but de restreindre l’accès aux clients qui connaissent le mot de passe correct. Pour définir le mot de passe d’un périphérique, suivez ces instructions :

    1. Sur l'ordinateur serveur, ouvrez MAX (double-cliquez sur l’icône Measurement & Automation située sur votre bureau).
    2. Double-cliquez sur Périphériques et interfaces pour afficher la liste des périphériques.
    3. Faites un clic droit sur le périphérique pour lequel vous souhaitez restreindre l’accès et sélectionnez Propriétés dans le menu pour ouvrir la fenêtre concernant la configuration des périphériques.
    4. Cliquez sur l’onglet Accès en mode déporté.
    5. Cliquez sur Définir le mot de passe pour ouvrir la boîte de dialogue concernant la définition du mot de passe et entrez ce dernier.

    Après avoir défini un mot de passe pour le périphérique, les clients RDA doivent spécifier le mot de passe correct pour utiliser le périphérique en mode déporté. Par défaut, le mot de passe est vide, indiquant ainsi que le périphérique n’est pas protégé par un mot de passe. Si un client est déjà en train dutiliser un périphérique RDA pendant que le mot de passe est changé, le client reçoit alors une erreur -10340 jusquà que le mot de passe correct soit défini. Pour définir le mot de passe correct, suivez ces instructions :

    1. Sur l'ordinateur client, ouvrez MAX (double-cliquez sur l’icône Measurement & Automation située sur votre bureau).
    2. Double-cliquez sur Périphériques et interfaces pour afficher la liste des périphériques.
    3. Faites un clic droit sur le périphérique pour lequel vous souhaitez restreindre l’accès et sélectionnez Propriétés dans le menu local pour ouvrir la fenêtre de configuration des périphériques.
    4. Entrez le mot de passe correct pour le périphérique lorsqu’on vous le demande..

    En plus des mots de passe définis sur le serveur pour chaque périphérique, un mot de passe de base peut également être défini pour éviter que des utilisateurs non autorisées ne puissent configurer le serveur RDA à distance. Pour définir ce mot de passe sur le serveur, ouvrez MAX sur le serveur et sélectionnez Outils»Configuration NI-DAQ»Définir le mot de passe de la configuration déportée. Une fois que ce mot de passe est défini, les clients qui souhaitent configurer le serveur en mode déporté par le biais de MAX (en cliquant sur Outils»Configuration NI-DAQ»Configuration DAQ déportée) doivent d’abord entrer le mot de passe correct. Les clients qui connaissent le mot de passe de base correct ont la possibilité de changer le mot de passe de base du serveur, ainsi que chacun des mots de passe correspondants aux différents périphériques, par le biais de la Configuration DAQ en mode déporté.

    Depuis NI-DAQ 6.7, vous pouvez définir une limite de temps pour le mode RDA (time out). Ceci est utile quand la connexion réseau est perdue alors que la machine cliente RDA attend une réponse de la machine serveur RDA. Sans le time out, l’ordinateur de votre client RDA attendrait indéfiniment, donnant l’impression que votre programme LabVIEW est bloqué. Grâce à la fonctionnalité time out, votre programme LabVIEW affichera -10340 noConnectError.

    La valeur par défaut de cette limite de temps est 240000 ms (4 minutes). Pour changer cette valeur, vous devez ajouter une valeur à une clé existant dans la base de registres. Pour cela, veuillez suivez les étapes suivantes :

    1. Exécutez regedit.exe ou regedt32.exe.
    2. Ouvrez HKEY_LOCAL_MACHINE»SOFTWARE»National Instruments»NI-DAQ»CurrentVersion»Configuration Manager.
    3. Sélectionnez New»DWORD Valuedans le menu Éditer.
    4. Nommez la nouvelle valeur ReceiveTimeoutValue.
    5. Double-cliquez sur ReceiveTimeoutValue pour faire apparaître une boîte de dialogue dans laquelle vous pourrez entrer la nouvelle valeur. Entrez la valeur de time out souhaitée en millisecondes. Assurez-vous que la Base est bien en mode Décimal. Une valeur nulle de time out indique un temps d'attente infini, ce qui correspond au comportement des versions de NI-DAQ antérieures à 6.7.
    6. Exécutez regedit.exe ou regedt32.exe.

    Gardez à l'esprit que les appels NI-DAQ pour lire ou écrire des données peuvent prendre un temps assez long à rendre la main, variant selon les fréquences d'acquisition ou de rafraîchissement et les quantités de données. Appeler le VI AI-Lire un buffer et lire 100 échantillons à une vitesse de 10 éch/s prend 10 s. Si la valeur de ReceiveTimeoutValue est inférieure à 10000 ms, vous recevrez l'erreur -10340 bien que votre connexion réseau fonctionne sans problème.

    Passage de valeurs 16 bits non signées et d’entiers 32 bits sous Visual Basic

    Paramètre 16 bits non signés : Le type INTEGER sous Visual Basic étant une valeur signée 16 bits, toute valeur supérieure à 32767 est considérée comme négative. Une solution à ce problème est décrite ci-après :

    1. Si la fonction est censée renvoyer un entier 16 bits non signé et que vous lisez une valeur négative en retour, ajoutez-y 65.536.

    Par exemple :

    Dim myUnsignedCnt As Long       

    Dim retCnt As Integer                             

    ‘(retCnt est la valeur renvoyée par la fonction appelée)

    ‘Vous pouvez appeler une fonction DAQ renvoyant une valeur 32 bits non signée et la stocker dans retCnt.

    if retCnt < 0 then

    myUnsignedCnt = CInt(retCnt) + 65536

    else

    myUnsignedCnt = retCnt

    end if

    1. Pour passer une valeur 16 bits non signée à une fonction dont le paramètre est de type 16 bits non signé, vous devez au préalable calculer la valeur à passer à la fonction.

    Par exemple :

    Dim myUnsignedCnt As Long       

    Dim ToPassVal As Integer

    ‘Vous calculez la valeur à passer à la fonction et vous la stockez dans myUnsignedCnt.

    if myUnsignedCnt > 32767 then

    ToPassVal = CInt(myUnsignedCnt - 65536)

    else

    ToPassVal = CInt(myUnsignedCnt)

    end if

    Paramètre 32 bits non signés : Le type LONG sous Visual Basic étant une valeur signée 32 bits, toute valeur supérieure à 2147483647 est considérée comme négative. Une solution à ce problème est décrite ci-après :

    1. Si la fonction est censée renvoyer un entier 32 bits non signé et que vous lisez une valeur négative en retour, ajoutez-y 4.294.967.296.

    Par exemple :

    Dim myUnsignedCnt As Double

    Dim retCnt As Long       

    ‘(retCnt est la valeur renvoyée par la fonction appelée)

    ‘Vous pouvez appeler une fonction DAQ renvoyant une valeur 32 bits non signée et la stocker dans retCnt.

    if retCnt < 0 then

    myUnsignedCnt = CInt(retCnt) +4294967296

    else

    myUnsignedCnt = retCnt

    end if

    2.      Pour passer une valeur 32 bits non signée à une fonction dont le paramètre est de type 32 bits non signé, vous devez au préalable calculer la valeur à passer à la fonction.

    Par exemple :

    Dim myUnsignedCnt As Double

    Dim ToPassVal As Long

    ‘Vous calculez la valeur à passer à la fonction et vous la stockez dans myUnsignedCnt.

    if myUnsignedCnt > 2147483647 then

    ToPassVal = CLng(myUnsignedCnt - 4294967296)

    else

    ToPassVal = CLng(myUnsignedCnt)

    end if
